# 4.branch 사본으로 돌려보고 master라는 메인에 넣기

# branch라는 사본만들기
git branch 이름 ,이후 동일하게 add와 commit을 하면된다

# 다시 메인으로
git switch master

# branch와 master의 conflict 충돌 
master와 branch에서 동일한 파일의 동일 위치 코드를 각각 따로수정하면 conflict가
발생하기 때문에 내가 직접 수동으로 수정해줘야함


# branch 삭제
merge 완료된 branch 삭제  git branch -d 브랜치명
merge 안한 branch 삭제 git branch -D 브랜치명

# rebase 쓰는 이유?
branch없이도 코드 잘짜는 고수 같음라고 할 뻔
간단하고 짧은 브랜치들은 이거 쓰면 깔끔해 보이기 때문임
다만, conflict 엔딩 많이 난다잉

# 워떠께 씀?
1.새로운 브랜치로 이동 
2.git rebase 중심브랜치명
3.중심 브랜치로 이동해서
4.git merge 새로운 브랜치명

# squash and merge 
git merhe --squash 새브런치

# git은 버전관리용으로
파일복구 ,commit 복구

파일 복구하는법  : git restore
git log --oneline
git restore 파일명 
git resotre --source 커밋 아이디
git restore --staged 파일명
git revert 커밋아이디1,커밋아이디2 -> 커밋 취소하기
git revert head 방금 커밋 취소하기
git reset --hard  커밋 아이디 : 이 커밋이 생성된 시점으로 모든걸 돌려버림
(협업시엔 사용금지)
git reset --soft 커밋아이디 : 변동사항 지우지 말고 스테이징해놓가
git reset --mixed 커밋아이디 : 변동사항 지우지 말고 unstage해놓기
